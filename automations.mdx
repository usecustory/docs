Automations help your team move from customer context to follow-through without recreating the same workflow by hand every time.

They are especially useful for smaller teams because they reduce coordination overhead in the moments that happen constantly:

- An item becomes important
- A signal moves in analytics
- A PR gets merged
- The team needs a weekly summary

## What automations can do

Custory automations can:

- Update journeys with AI instructions
- Create GitHub issues
- Create Jira issues
- Create Linear issues
- Send Slack messages
- Send Discord messages
- Pull analytics signals
- Fetch GitHub PR context

## Trigger types

Custory supports both scheduled and event-based triggers.

### Scheduled triggers

Use scheduled triggers for recurring review and reporting rhythms:

- Hourly
- Daily
- Weekly

### Event-based triggers

Use event-based triggers when the workflow should react the moment something important changes.

Supported examples include:

- Item created
- Item updated
- Item status changed
- Impact threshold crossed
- Effort threshold crossed
- Priority threshold crossed
- GitHub PR opened
- GitHub PR merged

## Filters and scope

Automations stay useful when they are scoped tightly.

Common scope and filter controls include:

- Journey
- Item group
- Status
- Impact
- Effort
- Changed fields
- Repository
- Base branch

This helps a small team avoid noisy workflows that generate more updates than value.

## AI-drafted actions

For messages, task creation, and some journey updates, Custory can use AI to generate a stronger output from the surrounding context.

That is useful when you want:

- Better issue titles and descriptions
- Clearer Slack or Discord summaries
- Context-aware journey updates

## Automation lifecycle

Automations move through statuses such as:

- Draft
- Active
- Paused
- Invalid when setup requirements are not met

That lifecycle is useful because it lets the team prepare a workflow before turning it on permanently.

## Manual runs and control

Custory supports operational control such as:

- Reviewing validation issues before activation
- Running scheduled automations manually
- Inspecting run history
- Pausing a workflow without deleting it
- Reactivating it later

This is important when the logic is sound, but the team wants to trial it carefully.

## Best first automations for lean teams

Most founder-led and product-led teams should start with one of these:

- A weekly journey pulse to Slack or Discord
- A priority-to-task handoff workflow
- A GitHub merged PR journey refresh
- A PostHog event drift workflow

Each of those solves a concrete coordination problem without requiring a large automation program.

## Common mistakes

### Automating before the journey data is current

Automations amplify the underlying data quality. If ownership, priority, or statuses are stale, the workflow will amplify stale context too.

### Starting with too many workflows

One reliable automation is better than six noisy ones the team ignores.

### Using automation where a simple operating habit would be enough

If the team only needs a weekly cleanup pass, build that habit first. Automate when repetition is real.
