Custory notifications help your team stay aware of the changes that matter without turning every update into another interruption.

## How notifications are organized

This page covers both the notification center and notification delivery setup.

It explains:

- what kinds of updates Custory can send
- where those updates appear
- how Slack, Discord, email, and in-app delivery work
- how to keep alerts useful instead of noisy

For smaller teams, the problem is usually not a lack of updates.

It is deciding which updates deserve:

- in-app visibility
- email
- Slack DM
- Discord DM

If everything interrupts the team, nobody trusts the alerts. If nothing gets routed well, important changes get missed.

## The notification center

Custory includes an in-app notification center accessible from the notification bell.

It is the default review surface for workspace activity.

The notification center is split into:

- Inbox
- Archive

Use **Inbox** for active notifications that still may require attention.

Use **Archive** to clear the active stream without losing the record from your personal history.

## What you can do in the notification center

From the notification center, you can:

- open related work directly
- mark notifications as read
- mark notifications as unread
- mark all inbox items as read
- archive items
- move archived items back to inbox

## Notification categories

Custory groups notification preferences into three main categories:

- Mentions
- Item updates
- System

### Mentions

Notifications when someone mentions you in a comment.

### Item updates

Notifications for changes on items you are assigned to, including comments, assignment changes, and status changes.

### System

Notifications for operational events such as invitation updates, join requests, [automation](/automations) failures, invalid automations, [integration](/integrations) errors, expiry, or [external task](/external-tasks) changes.

## Delivery channels

Custory supports these delivery channels:

- in-app
- email
- Slack DM
- Discord DM

In-app is the baseline. The other channels should be added intentionally for updates that deserve faster attention.

## How Slack and Discord DM setup works

Notification delivery happens at two levels:

- the workspace must have the Slack or Discord [integration](/integrations) connected
- each person must link their own Slack or Discord identity

### Slack DM setup

Slack DM delivery requires:

1. an active Slack workspace integration
2. your Slack account linked to your Custory account

From **Settings -> Notifications**:

1. generate a Slack link code
2. open the Custory app in Slack
3. send the generated code, or `link CODE`, in the Messages tab

### Discord DM setup

Discord DM delivery also requires:

1. an active Discord workspace integration
2. your Discord identity linked to your Custory account

From **Settings -> Notifications**:

1. generate a Discord link code
2. open a DM with the Custory bot in Discord
3. run `/link CODE`

## Link code behavior

Custory link codes are intentionally temporary.

Current behavior:

- codes expire after 15 minutes
- a code can only be used once
- you can regenerate a code
- you can clear an unused code
- you can unlink an already linked Slack or Discord account

## Recommended routing for smaller teams

Good default pattern:

- Mentions: in-app plus Slack or Discord DM
- Item updates: usually in-app only
- System: in-app plus email, and direct message only for high-responsibility roles

## Mistakes that make alerts noisy

<AccordionGroup>
  <Accordion title="Sending everything everywhere">
    This creates fatigue instead of clarity. Keep direct-message delivery for the updates that deserve interruption and let in-app notifications handle the rest.
  </Accordion>
  <Accordion title="Assuming workspace integration automatically links personal identities">
    Shared integration and personal linking are separate on purpose. Make sure both layers are configured before you expect Slack or Discord DMs to work reliably.
  </Accordion>
  <Accordion title="Using DM delivery for low-signal updates">
    Direct messages should be reserved for things someone should notice quickly. If a channel becomes noisy, the team will stop trusting it.
  </Accordion>
</AccordionGroup>

## What a good notification setup looks like

A good notification setup lets the team:

- catch up in-app
- respond quickly to high-signal alerts
- avoid noisy direct-message channels
- trace important system issues before they create bigger workflow drift
