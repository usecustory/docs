Use this guide when you want to go from "we just signed up" to "this is already helping the team make better decisions."

## What this is

This is the fastest path to a useful Custory setup.

It shows you how to:

- create one clean workspace
- choose the right first journey
- add enough structure to make the map useful
- invite the right people
- avoid turning setup into a big internal project

## Why it matters

Most teams do not fail because they cannot draw a journey.

They fail because they start too broad, invite too many editors, import too much material, and end up with a map nobody maintains.

Quickstart keeps the first week narrow so Custory becomes operational fast.

## Before you start

Have these decisions ready:

- The first journey you want to map
- The people who should edit it
- The source material you already have

Good first journeys for smaller teams:

- New user onboarding
- Activation
- Trial to paid
- Support escalation
- Renewal risk

Avoid starting with "the full customer lifecycle." That scope is too broad for an early working map.

## How it works

<Steps>
  <Step title="Complete onboarding">
    When you first sign in, Custory walks you through onboarding. The flow typically covers your role, use case, how you found Custory, workspace creation or workspace entry, and plan selection when billing setup is required.
  </Step>
  <Step title="Create your first workspace">
    Your workspace is the shared home for one product, one product area, or one operating team. Use separate workspaces only when you need real separation such as different products, clients, or business units.
  </Step>
  <Step title="Choose the first journey the team will actually maintain">
    From the dashboard, start with [Blank](/journeys), [Template](/journey-templates), or [AI-assisted import](/ai-journey-imports) based on how much source material you already have.
  </Step>
  <Step title="Build the map in the right order">
    Name the journey, add major stages, add steps inside each stage, attach the first useful evidence, and link the right persona if one matters.
  </Step>
  <Step title="Invite a small editor group">
    Start with the smallest useful group so the map stays consistent while the team builds its first working habits.
  </Step>
  <Step title="Connect only the tools that reduce real friction">
    Add the systems your team already checks every day instead of trying to connect everything in week one.
  </Step>
</Steps>

<Warning>
  If someone invited you to an existing workspace, use that route instead of creating a duplicate workspace.
</Warning>

## Build the map in the right order

Do not start by adding too many items.

Start in this sequence:

1. Name the journey clearly
2. Add the major stages
3. Add the steps inside each stage
4. Attach the first useful evidence
5. Link the right persona if one matters

For smaller teams, a strong first journey usually has:

- 4 to 7 stages
- clear customer-language step names
- enough detail to support decisions without becoming heavy to maintain

Better step names:

- Customer connects Slack
- Customer invites teammate
- Customer asks for help

Worse step names:

- Activation workstream
- PMF onboarding
- Support workflow layer

## 5. Add evidence, not decoration

The real value starts when you attach context to the journey.

Use items to capture:

- [Touchpoints](/touchpoints)
- [Insights](/insights)
- [Opportunities](/opportunities)
- [Solutions](/solutions)
- [Metrics](/metrics)

Open each item and add the material that helps the team act:

- description
- files or screenshots
- external links
- comments
- owners where relevant
- status and prioritization fields

If the journey only has boxes and labels, it will look organized without actually helping the team.

## 6. Invite a small editor group first

Most founder-led teams get the best results when they start with 2 to 5 editors, not the whole company.

Good early editors:

- founder or product lead
- one support or customer-facing teammate
- one engineering lead or product engineer
- one design or research owner if relevant

Use viewers for people who need visibility without changing the source of truth.

## 7. Connect only the tools that reduce real friction

Do not connect every integration during week one.

Start with the systems your team already checks every day:

- Slack or Discord for updates
- GitHub, Jira, or Linear for follow-up work
- Notion or Figma if they already hold journey-relevant source material

For a small team, one communication tool plus one delivery tool is often enough to make Custory feel operational.

## 8. Use AI where it saves real time

Custory AI is most useful when it helps you get through messy early work faster.

Use it to:

- [draft a first journey from source material](/ai-journey-imports)
- tighten stage and step wording
- summarize evidence
- draft follow-up tasks or updates
- review the journey for gaps

## Common mistakes

<AccordionGroup>
  <Accordion title="Starting too broad">
    Pick one journey that matters now. You can add more later.
  </Accordion>
  <Accordion title="Importing everything and refining nothing">
    AI import is a starting point. You still need to shape the output into a useful working map.
  </Accordion>
  <Accordion title="Over-inviting editors">
    Start with a small accountable group. Expand later.
  </Accordion>
  <Accordion title="Treating the journey like a static artifact">
    Custory works best when the journey changes with the product, not after the product.
  </Accordion>
</AccordionGroup>

## What good looks like

By the end of week one, you should have:

- one workspace
- one focused journey
- a small editor group
- real evidence attached to the map
- at least one review habit the team will actually keep
