Use this guide when you want to go from "we just signed up" to "this is already helping the team make better decisions."

Custory works best when you keep the first setup narrow:

- One workspace
- One high-value journey
- A small editor group
- Only the integrations you will actually use this week

That gives you a real operating system early instead of a large setup that nobody maintains.

## Before you start

Have these decisions ready:

- The first journey you want to map
- The people who should edit it
- The source material you already have

Good first journeys for smaller teams:

- New user onboarding
- Activation
- Trial to paid
- Support escalation
- Renewal risk

Avoid starting with "the full customer lifecycle." That scope is too broad for an early working map.

## 1. Complete onboarding

When you first sign in, Custory walks you through onboarding.

The flow typically covers:

- Your role
- Your use case
- How you found Custory
- Workspace creation or workspace entry
- Plan selection when billing setup is required

If you are creating the workspace yourself, finish onboarding before inviting the rest of the team. That keeps the structure clean and prevents people from landing in an empty account with no working context.

If someone invited you to an existing workspace, use that route instead of creating a duplicate workspace.

## 2. Create your first workspace

Your workspace is the shared home for one product, one product area, or one operating team.

Use separate workspaces only when you need real separation:

- Different products
- Different clients
- Different business units

Do not split workspaces just because one journey is different from another. That is usually what multiple journeys are for.

## 3. Create the first journey the team will actually maintain

From the dashboard, you can start a journey in three ways:

- [Blank](/journeys)
- [Template](/journey-templates)
- [AI-assisted import](/ai-journey-imports)

Choose the starting path based on the material you already have.

Use **blank** when:

- You already know the journey well
- You want full control over structure
- You do not want AI to infer the first draft

Use a **template** when:

- You want a proven starting structure
- The team is still learning how to map journeys
- You want momentum more than originality

Use **AI import** when:

- The source already exists in GitHub, Notion, Figma, or on your website
- You want Custory to turn existing material into a first draft
- The team would rather refine a draft than start from zero

## 4. Build the map in the right order

Do not start by adding too many items.

Start in this sequence:

1. Name the journey clearly
2. Add the major stages
3. Add the steps inside each stage
4. Attach the first useful evidence
5. Link the right persona if one matters

For smaller teams, a strong first journey usually has:

- 4 to 7 stages
- Clear customer-language step names
- Enough detail to support decisions, but not so much that updates become painful

Write step names from the customer's point of view.

Better:

- Customer connects Slack
- Customer invites teammate
- Customer asks for help

Worse:

- Activation workstream
- PMF onboarding
- Support workflow layer

## 5. Add evidence, not decoration

The structure matters, but the real value starts when you attach context to the journey.

Use items to capture:

- [Touchpoints](/touchpoints)
- [Insights](/insights)
- [Opportunities](/opportunities)
- [Solutions](/solutions)
- [Metrics](/metrics)

Open each item and add the working material that helps the team act:

- Description
- Files or screenshots
- External links
- Comments
- Owners where relevant
- Status and prioritization fields

If the journey only has boxes and labels, it will look organized without actually helping the team.

## 6. Invite a small editor group first

Most founder-led teams get the best results when they start with 2 to 5 editors, not the whole company.

Good early editors:

- Founder or product lead
- One support or customer-facing teammate
- One engineering lead or product engineer
- One design or research owner if relevant

Use viewers for people who need visibility without changing the source of truth.

If too many people can edit before the team has working habits, the map usually becomes inconsistent quickly.

## 7. Connect only the tools that reduce real friction

Do not connect every integration during week one.

Start with the systems your team already checks every day:

- Slack or Discord for updates
- GitHub, Jira, or Linear for follow-up work
- Notion or Figma if they already hold journey-relevant source material

For a small team, one communication tool plus one delivery tool is often enough to make Custory feel operational.

## 8. Use AI where it saves real time

Custory AI is most useful when it helps you get through messy early work faster.

Use it to:

- [Draft a first journey from source material](/ai-journey-imports)
- Tighten stage and step wording
- Summarize evidence
- Draft follow-up tasks or updates
- Review the journey for gaps

Leave memory on when you want AI to use workspace context from earlier work. Turn it off when you want a narrower answer for the current chat.

## 9. Establish one simple operating rhythm

Your first success does not come from filling every field. It comes from using the journey repeatedly.

A good starter rhythm for small teams:

1. Review the journey once a week in **Journey view**
2. Clean up stale items in **Table view**
3. Prioritize opportunities in **Impact vs effort**
4. Update linked work after decisions are made

If the map is not part of real planning, support review, or release follow-up, it will drift.

## Common early mistakes

### Starting too broad

Pick one journey that matters now. You can add more later.

### Importing everything and refining nothing

AI import is a starting point. You still need to shape the output into a useful working map.

### Over-inviting editors

Start with a small accountable group. Expand later.

### Treating the journey like a static artifact

Custory works best when the journey changes with the product, not after the product.

### Connecting too many tools too early

Only add integrations that reduce current friction.
