Custory integrations exist to reduce context drift.

The goal is not to connect everything. The goal is to connect the systems that already hold meaningful work so your team can move between journey context and execution without rebuilding the same explanation in every tool.

## Integration categories

Custory currently supports these broad groups:

- Product management: Jira, Linear
- Knowledge base: Notion, Google Drive
- Analytics and revenue: PostHog, Stripe
- Communication: Slack, Discord
- Developer tools: GitHub, MCP server
- Customer feedback: Intercom
- Collaborative editors: Figma, Miro

## What to connect first

For most founder-led and product-led teams, the best order is:

1. One communication tool
2. One delivery tool
3. One knowledge or context source
4. Analytics or revenue signals if they shape roadmap decisions often

That usually means something like:

- Slack or Discord
- GitHub, Jira, or Linear
- Notion or Figma
- PostHog or Stripe

## Communication integrations

### Slack

Use Slack to:

- Send automation updates to channels
- Link personal identities for direct-message notifications
- Work with Custory AI from collaboration threads

### Discord

Use Discord to:

- Send updates into channels and threads
- Link personal identities for direct-message notifications
- Work with Custory AI in conversation-heavy workflows

See [Slack and Discord AI threads](/slack-discord-ai-threads).

## Delivery integrations

### GitHub

Use GitHub when journey context should stay connected to issues, pull requests, and shipping activity.

### Jira and Linear

Use Jira or Linear when opportunities and solutions need structured execution follow-up in your team's delivery system.

### Notion as linked work

Notion can also support lighter-weight operational follow-up where a full engineering issue is not the right artifact.

See [External tasks and issues](/external-tasks).

## Knowledge and design integrations

### Notion and Google Drive

Use these when research notes, docs, or structured background material should inform journey work or AI prompts.

### Figma and Miro

Use these when visual flow context or collaborative board material should stay connected to the journey.

## Analytics and revenue integrations

### PostHog

Use PostHog when product usage changes should influence journey updates, insights, or automations.

### Stripe

Use Stripe when billing, payment failure, churn, or refund signals are important to the journey and should feed team awareness or automation logic.

## AI-related workflows

Integrations matter to AI because they expand the context Custory can work from.

That is useful when:

- Building a journey from existing source material
- Generating follow-up tasks from real workspace context
- Running Slack or Discord collaboration threads
- Using automation templates that rely on PostHog, GitHub, Jira, or Linear

## Integration management

Open **Manage Integrations** in a workspace to:

- Connect tools
- Reconnect expired integrations
- Review connection state
- Add manual credentials where required
- Request unsupported integrations

## Good integration hygiene

### Connect only the tools that remove real friction

More integrations are not automatically better.

### Set journey defaults where relevant

If one journey always routes work to the same repo, team, or project, set that default once instead of relying on memory.

### Keep delivery links close to item context

The integration is most valuable when it keeps the reason for the work visible, not just the task itself.
