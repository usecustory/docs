Use this page when you are evaluating whether Custory fits your team's security and privacy expectations.

This guide focuses on the controls that are clear in the product model and docs today: workspace boundaries, roles, integration scope, AI access patterns, and operational hygiene.

## What security means in Custory

For most lean teams, security in Custory starts with one practical question:

who can see the customer context, who can change it, and which outside tools can act on it?

Custory is built around scoped workspaces and explicit connections, so the main control points are:

- [workspace](/workspace) boundaries
- [member roles](/team-management)
- [integrations](/integrations)
- [notifications](/notifications)
- [MCP server](/mcp-server) access

## Workspace boundaries come first

The workspace is the main security boundary in Custory.

That matters because a workspace determines:

- which journeys live together
- which members can access them
- which integrations are shared
- which AI memory and MCP access stay scoped together

If two teams should not share customer context, they should not share the same workspace.

## Role-based access keeps editing tight

Custory separates people who administer the workspace from people who maintain the journey and people who only need visibility.

- `Owner`: manages workspace settings, members, billing, and destructive actions
- `Editor`: updates journeys, items, and operating workflows
- `Viewer`: follows the work without changing the source of truth

For most teams, the safest pattern is simple:

- keep owners limited
- give editor access only to people who actively maintain the system
- default everyone else to viewer

See [Manage your team](/team-management) for the role model.

## Integrations should be intentional, not automatic

Integrations are powerful because they connect the journey to analytics, delivery tools, chat, and AI workflows.

They also widen the operating surface, so connect only the tools that remove real friction.

Good integration hygiene looks like this:

- connect only the systems the team already relies on
- reconnect expired integrations intentionally
- keep delivery links close to the item context they came from
- review whether a workspace still needs every connected tool

See [Integrations](/integrations) for the full connected-stack model.

## AI access should stay scoped to real context

Custory supports AI inside the workspace and through [MCP](/mcp-server).

That is useful because AI can work from the same journey context as the team instead of from copied fragments. It also means access should be treated like any other system connection.

Good AI hygiene looks like this:

- use the correct workspace so AI retrieves the right context
- keep MCP keys limited to the clients and workflows that actually need them
- revoke and rotate access when a tool or person should no longer have it
- turn memory-aware workflows on only when prior context is actually helpful

See [AI as a workspace member](/ai-workspace-member), [Workspace memory](/workspace-memory), and [MCP server](/mcp-server).

## Privacy in practice

Custory works best when the team is deliberate about what customer material it puts into the workspace.

That usually means:

- attach only the evidence the team truly needs
- avoid dumping unrelated customer data into items
- keep comments, files, and links close to the customer moment they support
- remove access when someone no longer works on that product area

For privacy-sensitive evaluations, use a focused test workspace and verify the exact internal rules your team needs before broad rollout.

## What to confirm during evaluation

Some security reviews need details that belong in a buyer or implementation conversation, not in a generic docs page.

If those details matter for your team, confirm:

- environment or hosting expectations
- data-handling requirements
- retention expectations
- compliance or procurement requirements
- incident-response or audit expectations

This page is meant to explain the practical model, not to guess at controls that are not documented here.

## Common security mistakes to avoid

<AccordionGroup>
  <Accordion title="Using one workspace for teams that should be separated">
    If the access boundary is different, create a separate workspace. Do not rely on informal norms to enforce separation.
  </Accordion>
  <Accordion title="Giving editor access to everyone by default">
    More editors means more accidental drift and broader write access. Keep the accountable editor group small.
  </Accordion>
  <Accordion title="Connecting tools without a clear operating reason">
    Every integration should earn its place. If the team does not use the connected context or follow-up path, leave it disconnected.
  </Accordion>
</AccordionGroup>

## A secure setup in practice

A healthy Custory setup lets your team answer:

- who can access this workspace
- who can edit the operating system
- which outside tools are connected
- which AI clients can use the context
- whether the current setup still matches the real access boundary
