Version history lets your team move quickly in the editor without treating every big change as risky or irreversible.

It gives you a timeline of saved journey states and a safer way to recover when a workshop, cleanup pass, or AI-assisted edit goes too far.

## What version history helps you recover and review

This page covers both snapshot history and the collaboration patterns around it.

Use version history to:

- inspect earlier states
- preview before restoring
- label meaningful checkpoints
- recover from changes that moved too far or too fast

This is especially valuable for smaller teams because customer-journey work often changes in bursts:

- a workshop restructures the map
- a cleanup pass removes stale work
- a release changes the flow
- AI or collaborative editing moves faster than expected

Version history lets you inspect those changes without rebuilding the journey manually.

## Collaboration in the editor

Inside the journey editor, teams can work together with:

- presence avatars
- active editing cues
- cursor activity where relevant
- comments on items

Comments are especially useful because they keep discussion attached to the actual customer issue instead of scattering it across chat tools.

## What version history captures

Custory records snapshots from events such as:

- session start
- session end
- auto-save
- milestone saves
- manual saves

These snapshots create a recoverable timeline instead of one fragile current state.

## How to use it safely

The safest workflow is:

1. open version history
2. select the snapshot you care about
3. preview that version
4. confirm it is the state you want
5. restore only if needed

Do not restore based on timestamp alone.

## Label important versions

Use labels for moments such as:

- `Before onboarding rewrite`
- `Post workshop baseline`
- `After Q2 cleanup`
- `Before restoring shipped flow`

Labels matter because timestamps alone are easy to forget in fast-moving workflows.

## When version history is most useful

Version history is especially valuable for:

- weekly journey maintenance
- cross-functional workshops
- AI-assisted restructuring
- recovery after aggressive cleanup
- reconstructing decision context over time

## Recovery habits that cause problems

<AccordionGroup>
  <Accordion title="Never labeling meaningful states">
    Important versions become hard to find later. Label the checkpoints the team may actually need again, especially before large workshops, restructures, or cleanup passes.
  </Accordion>
  <Accordion title="Restoring without previewing">
    You increase the chance of bringing back the wrong state. Preview first whenever the restore matters so you do not replace good current work with an older version you only partly remembered.
  </Accordion>
  <Accordion title="Using comments outside the item or journey context">
    If the reasoning stays off-item, the next person has to reconstruct it from memory. Keep comments attached to the journey or item where the decision actually lives.
  </Accordion>
</AccordionGroup>

## What safe collaboration looks like

A healthy collaboration setup gives the team confidence to:

- work in the same journey together
- discuss changes where the work lives
- recover earlier states when needed
- move faster without fearing irreversible mistakes
